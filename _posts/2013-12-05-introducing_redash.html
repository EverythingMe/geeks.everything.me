tl;dr

We’ve been using Amazon Redshift as our analytics database and needed a better way to collaborate on queries, visualization and dashboards. So we built it.

Background

We're collecting events across client applications (e.g. Everything Home for Android) and server side components and needed a coherent and full analytics view and ability to query across those events in a single place. In the early days we chose Splunk to do our events analysis and analytics. Splunk is a great tool, but it’s meant for operational logs rather than analytics and we reached the point when Splunk just didn't cut it anymore. We could have probably made it work for us longer but it felt like the wrong place to put our time and effort into.

We switched to using Amazon Redshift as our analytics database, but have been left with no "UI" solution. At this point we tried several traditional BI suites, but we discovered bloated approach and unanswered technical challenges. The tools we tried failed to work with a columnar storage built to hold huge amounts of data in general or Redshift in specific, and to top that the process required to operate them felt slow and put the tool in the middle. I personally feel  that the world of BI has so much history (read: legacy) behind it which makes the default implementations and approaches very far from how we roll engineering and culture wise.

So our data sits in Redshift, and all you need is to know enough SQL to be dangerous and start pulling out insights, you start querying the data and voila you have nice dataset to play with in your favorite spreadsheet tool. Soon enough queries start flying around in email threads, people start sharing their unique and quirky ways to abuse sql and Redshift and attach spreadsheet files. At that point you lose your hands and legs trying to figure out which version of the query from the thread is the correct one.

We tried Chart.io, which was more inline with our needs, but even with it SQL queries were not first class citizens and it was missing collaboration features that were needed to compliment the way we worked.

As no ready-made solution was to be found, we decided to try and create our own solution at our 24 hours hackathon event (aka #hackAthing) - re:dash. re:dash was built under the assumption that we're not afraid to let users write raw queries to access their data, but we also want to give them rich tools to work with data and collaborate around it with their peers.

While querying the data requires you to be familiar with SQL, the idea is that other users will be able to participate by using existing datasets.

Current Status

re:dash consists of two parts:

  • Query Editor: think of JS Fiddle for SQL queries. It's your way to share data in the organization in an open way, by sharing both the dataset and the query that generated it. This way everyone can peer review not only the resulting dataset but also the process that generated it. Also it's possible to fork it and generate new datasets and reach new insights.
  • Dashboards/Visualizations: once you have a dataset, you can create different visualizations out of it, and then combine several visualizations into a single dashboard. Currently it supports bar charts, pivot table and cohorts.

This is the first release, which is more than usable but still has its rough edges and way to go to fulfill its full potential. The Query Editor part is quite solid, but the visualizations need more work to enrich them and to make them more user friendly.

It should be quite simple to install on your own server, but also on Heroku. And if you need help to get started, please contact us -- we’re happy to help.

Vision

Our vision is to create a tool that lets you easily create meaningful datasets, collaborate around it and share the information in a visual way.

The current re:dash state is just the first step in this direction, we're sharing its near future roadmap and would love to collaborate with others with similar needs as either users or contributors.
